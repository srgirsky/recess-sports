# CLAUDE.md — AI context for Recess Sports

Read this first. It's the fast on-ramp; deeper context is in `docs/OVERVIEW.md`, and human run/build/deploy steps are in `README.md`. **Keep all three current — see "Keeping docs current" at the bottom.**

## What this is (the one thing to understand)

A **free web baseball game for little kids (ages 4–8)**. You draft 9 of 30 neighborhood characters, then play a short game where you bat the top of each inning and pitch the bottom — one timing button for both. Balls in play become **live plays**: you steer a fielder with the pointer and charge throws on defense, and tap to send runners on offense (CLASSIC/KID MODE toggle on the title). The real product is **the 30 characters**: the game is a "voting machine" — every player draft pick is tallied, and pick rates reveal which kids should become toys/shows. So two jobs: (1) be genuinely fun, (2) log pick data. Design pillars: minimal reading, icon/voice-forward, short games.

## Stack

Phaser 3 · TypeScript · Vite · vitest. Static site, no backend, deployed free on **GitHub Pages** (auto-deploys on push to `main`; live at https://srgirsky.github.io/recess-sports/). All state (incl. pick tallies) is in `localStorage` for now.

## Architecture rules (follow these)

- **Pure game logic lives in `src/systems/` with NO Phaser imports.** Draft, at-bat resolution, innings, live plays, and pick logging are plain functions: state in → result out. They're unit-tested in `src/systems/logic.test.ts` + `liveplay.test.ts`. This is the most important rule — it keeps the tricky logic testable and lets the render loop stay dumb.
- **Scenes are the thin view layer.** They read input, call `systems/` reducers, and animate the result. A scene should never *decide* game outcomes — it plays back what a reducer returned. (E.g. baserunning animation on walks is driven by `ApplyResult.movements` from `inning.ts`, so it can't desync from the real base state.)
- **Balls in play run through the live-play sim** (`systems/liveplay.ts`): `resolveContact` (atbat.ts) turns a contact band into a trajectory, then GameScene's `update()` steps `stepLivePlay(state, inputs, dt)` every frame and positions everything the sim owns (ball, fielders, runners) **directly — never with tweens**. Outs emerge from the race. Kid mode: fly-outs + ball-beats-runner-to-the-bag only; caught fly returns runners free. Main mode (`params.manualBaserunning`): real rules — force-outs only when forced, tags for unforced runners (carrier within `RUN2.TAG_RADIUS` of an off-bag runner; rundowns emerge), tag-ups with doubling off, sac flies (a `CATCH_GATHER_MS` beat after catches keeps them winnable), per-runner send/hold via `LiveInputs.sendRunner/holdRunner` (tap a base ahead = send, behind = turn back). CPU policies for the non-human side live inside the sim. `finishLivePlay` → `applyLivePlay` (inning.ts) folds the result back into the half-inning. Field coordinates shared by sim + renderer live in `systems/geometry.ts`. The scene draws through the 3/4 camera in `art/projection.ts` (`project`/`unproject`/`depthScale`): the sim and ALL tests stay in flat logical space; only the render/input membrane projects (pointer events unproject on the way in). Never import projection from `systems/`. On top of it, GameScene runs a **two-view camera choreography** (`config.CAMERA`): a zoomed batting/pitching close-up on the plate–mound corridor for every pitch, snapping wide the moment a ball is live, runners move, or a steal race starts; HUD chrome renders on a dedicated UI camera via `pinUI()` (plus a corner mini-diamond inset: base state + the STEAL! chips) and never zooms; the pinned scoreboard strip (score panel / inning pill / labeled B-S-OUT pips with a change pulse + the HUD-anchored BALL!/STRIKE!/FOUL! umpire calls) lives in `scenes/ui/Scoreboard.ts`. Game mode (CLASSIC 'main' / KID MODE 'kid', persisted, main is the default) merges `config.LIVE` × `config.MODES[mode].live` via `systems/mode.ts`; `MODES[mode].features` flags gate the Backyard-style mechanics (all off in kid mode).
- **Character art is generated once in `BootScene`** as modern toy-brand SVG (no image files): bold navy outlines with soft **gradient volume** inside them. **Lighting convention: one warm key light from the UPPER-LEFT, everywhere.** Shadows are cool (mixed toward navy via `darken`), highlights warm (`lighten`). `gradientDefs` in `CharacterArt.ts` builds per-document gradients from the kid's palette (radial `skinG`/`hairG` light-stop biased upper-left, linear `jerseyG`/`pantsG`); filled regions use them via `Ctx.gSkin`/`gJersey`/`gPants`. **Never put a gradient on a stroke** — `objectBoundingBox` gradients are undefined on zero-width bboxes (straight limbs); `capsule()` limbs keep their offset highlight-stroke instead. `hair()` takes pre-computed fill strings (deriving a shade from `'url(#...)'` NaNs — the art test catches it). The field agrees: the sun draws top-left, the mound is a lit dome, `shadeInt`/`lightenInt` in GameScene mirror the same mix. Keep any new art on this light or kids will look pasted on. `art/CharacterArt.ts` builds an SVG string from `VisualParams` (skin/hair/hairColor/uniform/accessory + `expression`/`bodyType`/`freckles`) **and a `pose`** (`stand` front view, `run1`/`run2` side-view run cycle drawn facing right — flipX for left, `cheer` arms-up, `bat` side-view batting stance with the bat baked in (faces right — no flip at the plate), `windup` front-view leg-lift pitch, `ready` fielding crouch, `slide` low side-view with dust); `art/textureFactory.ts` turns each into a Phaser texture (3× viewBox) keyed via `poseKey(id, pose)` — plain `id` for stand, `id:run1` etc. Animate runs by swapping textures with `anim.runCycle()`; put `anim.groundShadow()` under kids. Dev art gallery: press **G** on the Schoolyard, then **P** cycles poses, **A** animates the run (pick rates: **D**).
- **All "feel" tunables live in `src/config.ts`** (timing windows, pitch speed, innings, shake, runner speed, audio volume, the `LIVE` live-play block, the `FLOW` pacing block, and the `MODES` kid/main table: live-sim multipliers + feature flags). Change feel there, not inside scenes. `FLOW` owns every between-moments beat (next pitch, next batter, half start, banner holds) — GameScene `delayedCall`s must use it, never literal ms; invariant: a banner's hold ≤ the FLOW beat that follows it, so calls finish before the next pitch.
- **Reusable juice** (shake/burst/floating text/confetti) is in `src/ui/effects.ts`; **character animation** helpers (idle bob, squash-hop with `onDone`, pop-in, `enterFrom` staggered reveals, `pulse` attention loops, `groundShadow`, `runCycle` two-frame texture animation) are in `src/ui/anim.ts`; **sound** (free, code-synthesized incl. the recess `bell()`) is in `src/systems/audio.ts`. Character motion combines pose-texture swaps (`runCycle`) with procedural tweens — e.g. GameScene draws a swinging **bat** prop and a pitcher **wind-up**. The Schoolyard is title + draft in one continuous scene: PLAY rings the bell, doors burst open, kids stream to the brick wall (the stream-out is a real crowd sim — `systems/crowd.ts`, stepped from `SchoolyardScene.update()` with pairwise separation so kids never overlap; feel knobs in `config.CROWD`), tap-to-inspect stat cards, picked kids run to their team's pennant, both teams cheer. Other feel knobs live in `config.ANIM`.
- **Shared UI kit** is `src/ui/theme.ts`: the brand `FONT`, `OUTLINE`, and rounded-outlined-with-shadow helpers `panel()`, `ribbon()`, `pill()`, `heading()`. Use these (not raw `add.rectangle`) for any UI chrome — Phaser rectangles can't round corners. Buttons/cards (`ui/Button.ts`, `ui/CharacterCard.ts`) are built on it. The font is self-hosted Fredoka (`public/fonts/fredoka.woff2`, `@font-face` in `index.html`), awaited in `BootScene` before Title.

## Gotchas (things that will bite you)

- **Background tabs pause the game.** Browsers stop `requestAnimationFrame` when the tab isn't foreground, which freezes Phaser's clock — timers (AI pick, pitch) don't fire and input drops. Not a bug; playtest with the tab focused. For automated/headless visual checks, drive `window.__game` and force swings via `gameScene.resolvePlayerSwing(band, false)` (top half) or pitches via `gameScene.resolvePlayerPitch(band)` (bottom half — wait for `phase === 'aiming'`). **Pumping the clock correctly matters**: call `game.loop.step(t)` with monotonically increasing `t` (always > `game.loop.time` — screenshots can advance the real clock behind your back), NOT `game.step(t, delta)` (leaves `loop.delta` stale). And Phaser's tweens run on **wall-clock `Date.now()`**, not the game clock — to fast-forward tweens headlessly you must also rewind each active scene's `scene.tweens.startTime` by your step size per pump. Timers (`delayedCall`) follow the loop clock; tweens follow `Date.now()` — pump both or animations desync from logic.
- **Live plays are stepped from `GameScene.update()` on the game-loop clock — never tween anything the sim owns** (the live ball, fielders, runner tokens mid-play). Tweens are for chrome only (rings, meters, banners, exit fades). Headless: drive a live play with `gameScene.setLivePointer(x, y)`, `commandThrow(base, power)`, `commandRun()`, and inspect `getLivePlay()` while pumping the loop.
- **The Schoolyard stream-out is sim-owned, like live plays.** `SchoolyardScene.update()` steps `stepCrowd` only while `phase === 'cutscene'` (the scene lives through the whole draft — a stray step would fight idleBob/inspect tweens on the same containers). Never tween a kid's `root` mid-stream. The render-side run bob writes `kid.img.y` (the child inside the container) — zero it at the settle boundary before `idleBob` captures its baseline, or kids idle-bob around an offset. Kid draw order is `yardDepth(y)` (y-sorted, ≈11–34), not fixed constants.
- **Stop `runCycle` timers before destroying their sprite.** The cycle swaps textures on a repeating scene timer; destroying the image while the timer lives used to crash the whole game loop (`setTexture` on a dead sprite). `runCycle` now self-guards on `img.active`, but still `cycle.stop()` in teardown paths — and never re-start a cycle for a runner whose `done` is set (their exit fade owns the sprite).
- **Phaser polygon points must be 0-based (no negative coords).** `add.polygon` computes its display origin from the AABB but does NOT normalize negative point coords, so negatives get shifted twice and the shape renders far from where you put it (this misplaced the infield dirt over the stands once). Author polygon point lists starting at 0,0. **`add.triangle` fills are similarly unreliable — draw filled/stroked shapes with Graphics** (`fillTriangle`/`strokeTriangle`), like `ui/theme.ts` does for rounded chrome. Also: shape color arrays need integer indexes — `bunt[(x / 56) % len]` silently yields `undefined` (invisible fill) when `x/56` is fractional; use `Math.floor`.
- **SVG textures need base64 data URIs.** Phaser's `load.svg` calls `atob` on data URIs — they must be base64 (`textureFactory.ts` does this), not URL-encoded, or it throws `InvalidCharacterError`.
- **`textures.getTextureKeys()` is polluted** by Phaser Text objects (GUID keys). Don't treat texture keys as character ids — use `ROSTER` from `data/characters.ts`.
- **All pose art must bottom out on the same ground line** (`GROUND = 248` in `CharacterArt.ts`): sprites use `setOrigin(0.5, 1)` and swap pose textures in place, so a pose whose lowest ink sits higher/lower makes the kid visibly pop up or sink when a run cycle starts.
- **Phaser camera zoom ignores `setScrollFactor`** — HUD can't be pinned that way. GameScene runs a second UI camera: every screen-anchored element must go through `pinUI()` (it also routes container children — the input hit-tester checks them individually) or it will zoom with the field; an `addedtoscene` hook auto-hides everything else from the UI cam. Read world pointer coords via `cameras.main.getWorldPoint(p.x, p.y)`, NOT `pointer.worldX/worldY` — Phaser computes those against the topmost camera, which is the unzoomed UI cam.
- **Camera effects (`cam.pan`/`zoomTo`) need exact EaseMap keys** like `'Sine.easeInOut'` — tweens fuzzy-match the `'Sine.inOut'` shorthand, but camera effects don't resolve it, leave `this.ease` undefined, and crash the game loop on the next update.
- **Audio needs a user gesture** to start (browser autoplay policy). It's unlocked on the Title PLAY click (`audio.unlock()`); all audio calls no-op before that or when muted, so they're always safe to call.
